# Student Data Access Request Procedure (AC-PROC-005)

## 1. Purpose

This procedure provides step-by-step instructions for requesting, approving, and provisioning access to student data, ensuring compliance with AC-POL-003 and FERPA legitimate educational interest requirements.

## 2. Scope

This procedure applies to all workforce members requesting or approving access to student data.

## 3. Procedure

### 3.1 Requesting Access

#### 3.1.1 Submitting Access Request

**When:** Workforce member needs access to student data for job function.

**Steps:**

1. **Identify access needed**
   - Specific data categories (student records, usage data, etc.)
   - School(s) whose data is needed
   - Access level (read, write, admin)
   - Duration (temporary or permanent)

2. **Document business justification**
   - Job function requiring access
   - Specific tasks requiring data
   - Why this access level is needed

3. **Submit request**
   - Use access request system/form
   - Include all required information
   - Route to appropriate approver

4. **Await approval**
   - Do not access data until approved

### 3.2 Approving Access

#### 3.2.1 Standard Access Approval

**When:** Manager receives access request for direct report.

**Steps:**

1. **Review request**
   - Is access needed for job function?
   - Is scope appropriate (not excessive)?
   - Is duration appropriate?

2. **Verify legitimate educational interest**
   - Is data needed for contracted educational service?
   - Would school authorize this access?

3. **Approve or deny**
   - Approve: proceed to provisioning
   - Deny: document reason; notify requestor

4. **Document approval**
   - Record in access request system
   - Retain for audit

#### 3.2.2 Elevated Access Approval

**When:** Request requires elevated access (broad data access, sensitive data).

**Steps:**

1. **Manager approval**
   - Complete standard approval (Section 3.2.1)

2. **Privacy Officer review**
   - Student Data Privacy Officer reviews elevated requests
   - Assesses legitimate educational interest
   - Evaluates necessity and scope

3. **Additional approval**
   - Privacy Officer approves or denies
   - May require access restrictions

4. **Document chain of approval**
   - All approvals recorded

### 3.3 Provisioning Access

#### 3.3.1 Granting Access

**When:** Access request is approved.

**Steps:**

1. **Verify approval**
   - Confirm all required approvals in place
   - Check approval is current (not expired)

2. **Provision access**
   - Grant access per approved scope
   - Assign appropriate role
   - Limit to approved schools/data

3. **Set expiration** (for temporary access)
   - Configure access to expire
   - Set reminder for review

4. **Notify requestor**
   - Confirm access granted
   - Provide any usage guidance

5. **Document provisioning**
   - Record access granted
   - Date, scope, approvals

### 3.4 Access Review

#### 3.4.1 Periodic Access Review

**When:** Quarterly or as policy requires.

**Steps:**

1. **Generate access report**
   - List personnel with student data access
   - Access scope and level
   - Last access date

2. **Review each access**
   - Is person still employed?
   - Is access still needed for role?
   - Is scope still appropriate?

3. **Identify changes needed**
   - Remove access no longer needed
   - Reduce excessive access
   - Flag for termination if role changed

4. **Execute changes**
   - Remove/reduce access as identified
   - Notify affected personnel

5. **Document review**
   - Record review date, reviewer, findings
   - Retain for audit

#### 3.4.2 Role Change Review

**When:** Employee changes roles.

**Steps:**

1. **Notify access team**
   - HR/manager notifies of role change

2. **Review current access**
   - What access does person have?
   - What access does new role require?

3. **Adjust access**
   - Remove access not needed for new role
   - Grant new access per new role (with approval)

4. **Document changes**
   - Record access adjustments
   - Date of role change

### 3.5 Access Termination

#### 3.5.1 Employee Termination

**When:** Employee leaves company.

**Steps:**

1. **Receive termination notice**
   - HR notifies of termination

2. **Revoke access immediately**
   - Disable all student data access
   - Within 1 business day

3. **Disable accounts**
   - Disable system accounts
   - Remove from access groups

4. **Document termination**
   - Record access removal
   - Date of termination

#### 3.5.2 Contractor/Vendor Termination

**When:** Contractor engagement ends.

**Steps:**

1. **Receive end notice**
   - Contract owner notifies of end

2. **Revoke access**
   - Remove all student data access
   - Per contract timeline

3. **Confirm removal**
   - Verify no access remains
   - Document confirmation

### 3.6 Emergency Access

#### 3.6.1 Emergency Access Request

**When:** Urgent situation requires immediate access.

**Steps:**

1. **Identify emergency**
   - Critical support issue
   - Security incident
   - Urgent school request

2. **Request emergency access**
   - Contact IT/Security
   - Describe emergency and access needed

3. **Verbal approval**
   - Manager or Privacy Officer approves verbally
   - Document approval immediately after

4. **Grant temporary access**
   - Time-limited (auto-expire)
   - Minimum necessary scope

5. **Enhanced logging**
   - All actions logged
   - Monitor for anomalies

6. **Post-emergency review**
   - Review access use
   - Formalize approval retroactively
   - Revoke access when emergency ends

### 3.7 Audit Logging

#### 3.7.1 Access Log Requirements

**When:** All student data access.

**Steps:**

1. **Log all access**
   - User identity
   - Timestamp
   - Data accessed
   - Action (read, update, delete)
   - School context

2. **Protect logs**
   - Logs are tamper-resistant
   - Retained per policy

3. **Review logs**
   - Automated anomaly detection
   - Periodic manual review

4. **Provide to schools**
   - Upon school request
   - For their FERPA compliance

## 4. Escalation

Escalate to Student Data Privacy Officer when:
- Unusual access request
- Suspected unauthorized access
- Access dispute
- Audit finding

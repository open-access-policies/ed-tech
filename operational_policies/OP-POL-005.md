# School District Onboarding Policy (OP-POL-005)

## 1. Objective

This policy establishes the requirements for onboarding new school and district customers. It ensures that appropriate data protection agreements are in place, data handling requirements are understood, and **[Company Name]** is properly designated as a school official before receiving any student data.

## 2. Scope

This policy applies to:
- All new K-12 school and district customers
- All new higher education institution customers
- All contract renewals with material changes
- All workforce members involved in sales and implementation

## 3. Policy

**[Company Name]** does not receive or process student data until appropriate agreements are executed and the customer is properly onboarded with all required data protections in place.

### 3.1 Pre-Sale Requirements

**3.1.1 Initial Customer Qualification**

Before engaging with potential customers:
- Identify if customer is K-12, higher education, or mixed
- Determine applicable state(s) for compliance
- Confirm products/services align with customer needs
- Prepare relevant compliance documentation

**3.1.2 Security and Privacy Questions**

**[Company Name]** responds to customer security questionnaires, including:
- Common questionnaire formats (HECVAT, EDUCAUSE, state-specific)
- Privacy policy and terms of service
- SOC 2 report summary or full report (under NDA)
- VPAT for accessibility compliance
- Data handling and security practices

**3.1.3 Compliance Documentation Package**

Provide prospective customers with:
- Privacy policy (education-specific)
- Standard DPA or willingness to sign customer's DPA
- VPAT (accessibility documentation)
- Security overview or SOC 2 summary
- Insurance certificates (upon request)

### 3.2 Agreement Requirements

**3.2.1 Data Processing Agreement (DPA)**

Before receiving student data, execute a DPA that includes:

| Required Element | Description |
|------------------|-------------|
| **Services description** | Clear description of educational services |
| **Data elements** | Types of student data to be processed |
| **Permitted uses** | How data may be used (educational purposes only) |
| **Prohibited uses** | No sale, no advertising, no profile building |
| **Security requirements** | Encryption, access controls, incident response |
| **Subcontractor disclosure** | List of subcontractors with data access |
| **Breach notification** | Notification timeline and process |
| **Data retention/deletion** | Contract-limited retention; deletion procedures |
| **Parent rights** | Support for access, correction, deletion requests |
| **Audit rights** | School's right to audit compliance |
| **Governing law** | Applicable state law and jurisdiction |

**3.2.2 DPA Acceptance Hierarchy**

| Priority | Agreement Type |
|----------|---------------|
| 1 | Customer's DPA (if acceptable after legal review) |
| 2 | State-standard DPA (SDPC, CA-NDPA, etc.) |
| 3 | **[Company Name]** standard DPA |

**3.2.3 State-Specific Requirements**

For customers in states with specific requirements:

| State | Additional Requirements |
|-------|------------------------|
| **New York** | Parents' Bill of Rights attachment; supplemental information |
| **California** | SOPIPA attestation; CSPC compliance if applicable |
| **Colorado** | Data element inventory; security plan reference |
| **Connecticut** | Specific security attestations |
| **Illinois** | SOPPA compliance attestation |

See PRV-POL-003 and ANNEX-003 for complete state requirements.

**3.2.4 Agreement Review**

All customer agreements reviewed by:
- Legal/Compliance (for legal sufficiency)
- Student Data Privacy Officer (for privacy terms)
- Security (for security requirements)

### 3.3 Implementation Requirements

**3.3.1 Implementation Kickoff**

Before processing student data:
1. Executed DPA on file
2. Customer contacts identified and documented
3. Data handling requirements confirmed
4. Technical integration approach defined
5. Training needs identified

**3.3.2 Data Inventory**

Document student data to be received:
- Data elements by category (required, optional)
- Data sources (rostering, SIS integration, manual entry)
- Data flow diagram
- Retention requirements

**Template:** See ANNEX-004 (Student Data Inventory Template)

**3.3.3 Technical Setup**

| Activity | Requirement |
|----------|-------------|
| **Account provisioning** | Secure administrator account creation |
| **Integration setup** | Secure connection to customer systems (SSO, rostering) |
| **Data import** | Secure data transfer methods |
| **Access configuration** | Role-based access per school requirements |
| **Feature configuration** | Privacy settings per customer preferences |

**3.3.4 Rostering and Data Import**

For automated rostering (Clever, ClassLink, OneRoster):
- Verify integration is properly authorized by school
- Confirm data scopes match DPA
- Test data flow before full deployment
- Document roster fields mapped

For manual data import:
- Secure file transfer method
- Data validation before import
- Confirmation of successful import
- Secure deletion of import files

### 3.4 Training Requirements

**3.4.1 Customer Training**

Provide training to school/district staff on:
- Product features and usage
- Privacy settings and controls
- Reporting and analytics
- Support and escalation procedures
- Student and parent-facing features

**3.4.2 Internal Handoff**

Ensure internal teams understand:
- Customer-specific requirements
- State compliance requirements
- DPA terms and restrictions
- Support contact procedures

### 3.5 Documentation Requirements

**3.5.1 Customer Record**

Maintain for each customer:
- Executed contracts and DPAs
- Data inventory
- Authorized contacts list
- Configuration and feature settings
- Training completion records
- Compliance attestations

**3.5.2 FERPA Annual Notification Support**

Provide schools with:
- Information for their annual FERPA notification
- Template language for school official disclosure
- Company description for notification purposes

See STU-POL-002 Section 3.4 for template language.

### 3.6 Renewals and Changes

**3.6.1 Contract Renewals**

At renewal:
- Review for DPA updates needed (new state laws, changed practices)
- Confirm subcontractor list is current
- Update security documentation if changed
- Re-execute DPA if material changes

**3.6.2 Material Changes**

If **[Company Name]**'s practices materially change:
- Notify affected customers
- Offer DPA amendment if needed
- Obtain consent for expanded data use (if applicable)
- Document changes

**3.6.3 Customer-Requested Changes**

When customers request changes:
- Evaluate feasibility
- Document in customer record
- Update configuration as agreed
- Confirm implementation with customer

### 3.7 Onboarding Checklist

**3.7.1 Pre-Contract**

- [ ] Customer qualified (K-12, higher ed, state)
- [ ] Security questionnaire completed
- [ ] Compliance documentation provided
- [ ] Pricing and terms agreed

**3.7.2 Contracting**

- [ ] Service agreement executed
- [ ] DPA executed
- [ ] State-specific addenda (if applicable)
- [ ] Authorized contacts documented

**3.7.3 Implementation**

- [ ] Kickoff meeting completed
- [ ] Data inventory documented
- [ ] Technical integration configured
- [ ] Data import/rostering completed
- [ ] User accounts provisioned
- [ ] Training delivered

**3.7.4 Go-Live**

- [ ] Customer accepts implementation
- [ ] Support handoff completed
- [ ] Documentation provided to customer
- [ ] Customer record complete in CRM

### 3.8 Failed Onboarding

**3.8.1 Unable to Reach Agreement**

If parties cannot agree on terms:
- Document objections and attempts to resolve
- Offer alternative solutions if possible
- Decline engagement if terms are unacceptable
- Retain records of declined engagement

**3.8.2 Customer Cancellation During Onboarding**

If customer cancels before go-live:
- Delete any student data received
- Return data to customer if requested
- Document cancellation and data disposition
- Provide deletion confirmation

## 5. Definitions

| Term | Definition |
|------|------------|
| **DPA** | Data Processing Agreement governing student data handling |
| **Onboarding** | The process of setting up a new customer to use **[Company Name]**'s services |
| **Rostering** | Automated provisioning of student accounts from school systems |
| **SDPC** | Student Data Privacy Consortium, which maintains standard DPA templates |
| **Go-Live** | The point at which a customer begins active use of the service |
| **Implementation** | The technical setup and configuration process for a new customer |

| Role | Responsibility |
|------|---------------|

| **Legal/Compliance** | Review DPAs; advise on state requirements; approve agreements |

| **Implementation** | Execute technical setup; train customers; complete documentation |
| **Customer Success** | Manage ongoing relationship; handle renewals and changes |
| **Engineering** | Enable integrations; support technical configuration |

# Parent Rights Request Procedure (STU-PROC-003)

## 1. Purpose

This procedure provides step-by-step instructions for processing requests from parents and eligible students to access, review, correct, or delete their education records, in compliance with FERPA, COPPA, and state laws.

## 2. Scope

This procedure applies to Customer Success, Legal, and Engineering teams who receive and process parent rights requests.

## 3. Procedure

### 3.1 Receiving Requests

#### 3.1.1 Request Intake

**When:** A rights request is received from a school (forwarding parent request) or directly from a parent.

**Steps:**

1. **Log the request**
   - Date and time received
   - Source (school or direct)
   - Type of request (access, correction, deletion, opt-out)
   - Requestor information

2. **Identify the student**
   - Student name
   - School/district
   - Any verification information provided

3. **Classify request type**
   - Access/review request → Section 3.2
   - Amendment/correction request → Section 3.3
   - Deletion request → Section 3.4
   - Opt-out request → Section 3.5

4. **Set response deadline**
   - Initial response: **[Number, e.g., 5]** business days
   - Completion per request type (see sections below)

5. **Assign to handler**
   - Route to appropriate team member
   - Ensure tracking in request system

### 3.2 Access/Review Requests

#### 3.2.1 Processing Access Requests

**When:** Parent or school requests to review student's education records.

**Steps:**

1. **Verify request source**
   - If from school: Verify authorized school contact
   - If direct: Verify identity (government ID, knowledge-based verification)

2. **Compile responsive records**
   - Query all systems for student's records
   - Include all education records directly related to student
   - Exclude records of other students
   - Format in readable format

3. **Review for completeness**
   - All relevant records included
   - No third-party student information
   - Data elements explained if needed

4. **Prepare response**
   - Cover letter explaining contents
   - Records in accessible format (PDF, CSV as appropriate)
   - Explanation of data elements (if requested)

5. **Deliver to school** (if school-forwarded request)
   - Transmit records to school contact
   - School provides to parent per FERPA
   - Confirm delivery

6. **Deliver directly** (if direct relationship)
   - Verify identity before delivery
   - Use secure transmission method
   - Confirm receipt

7. **Complete within timeline**
   - Initial response within **[Number, e.g., 5]** business days
   - Full response within 45 days (FERPA requirement)

8. **Document completion**
   - Record response date and contents
   - Close request in tracking system

### 3.3 Amendment/Correction Requests

#### 3.3.1 Processing Amendment Requests

**When:** Parent or school requests correction of inaccurate education records.

**Steps:**

1. **Verify request source**
   - Confirm from authorized school contact or verified parent

2. **Identify data to be corrected**
   - Specific data element(s)
   - Current value
   - Requested corrected value
   - Reason for correction

3. **Evaluate request**
   - Is the data factually incorrect?
   - Is correction appropriate (not grades/evaluative content)?
   - Does school direct the correction?

4. **Make correction** (if approved)
   - Update data in all systems
   - Retain record of original value
   - Document correction made

5. **Deny correction** (if not approved)
   - Document reason for denial
   - Notify school of denial and reason
   - School handles any appeal per FERPA

6. **Confirm completion**
   - Notify school/requestor of outcome
   - Complete within **[Number, e.g., 10]** business days

7. **Document**
   - Record request and outcome
   - Retain correction log

### 3.4 Deletion Requests

#### 3.4.1 Processing Deletion Requests

**When:** Parent, student, or school requests deletion of student data.

**Steps:**

1. **Verify request source**
   - School direction required for school-contracted services
   - Direct parent request valid for COPPA (under 13) or home-use products

2. **Confirm scope of deletion**
   - All data or specific elements?
   - Immediate deletion or at contract end?
   - Exceptions (legal hold, retention requirements)?

3. **Check for exceptions**
   - Legal hold → cannot delete; notify requestor
   - Contractual retention requirement → coordinate with school
   - Aggregated/de-identified → explain not subject to deletion
   - Legal requirement → retain required data; delete remainder

4. **Execute deletion**
   - Delete from primary systems
   - Delete from backups (where feasible)
   - Confirm deletion from subcontractor systems

5. **Terminate access** (if data essential to service)
   - Disable student account
   - Notify school of service termination

6. **Confirm deletion**
   - Notify school/requestor of completion
   - Provide deletion confirmation
   - Complete within **[Number, e.g., 30]** days

7. **Document**
   - Record deletion request and execution
   - Retain deletion log per retention schedule

### 3.5 Opt-Out Requests

#### 3.5.1 Processing Opt-Out Requests

**When:** Parent requests to stop further data collection.

**Steps:**

1. **Verify request**
   - Confirm from school or verified parent

2. **Implement opt-out**
   - Stop further collection for the student
   - Configure systems to prevent new data capture
   - Implement within **[Number, e.g., 5]** business days

3. **Assess service impact**
   - Can service continue without data collection?
   - If not, what are consequences?

4. **Notify of impact**
   - Inform school/parent of any service limitations
   - Confirm understanding of consequences

5. **Coordinate with school**
   - School determines if student continues using service
   - Implement school's direction

6. **Document**
   - Record opt-out and implementation
   - Note any service changes

### 3.6 Consent Withdrawal

#### 3.6.1 Processing Consent Withdrawal

**When:** Parent withdraws previously given consent.

**Steps:**

1. **Verify withdrawal**
   - Confirm from school or verified parent

2. **Stop processing**
   - Cease further data collection/processing
   - Implement within **[Number, e.g., 5]** business days

3. **Process deletion**
   - Follow deletion procedure (Section 3.4)
   - Unless legally required to retain

4. **Terminate service**
   - If data essential, terminate access
   - Notify school of termination

5. **Confirm withdrawal effect**
   - Applies prospectively
   - Prior lawful processing not affected

6. **Document**
   - Record withdrawal and actions taken

## 4. Escalation

### 4.1 Escalation Triggers

Escalate to Student Data Privacy Officer when:
- Request is unclear or unusual
- Legal hold may apply
- Request involves legal dispute
- Requestor disputes response
- Timeline cannot be met

### 4.2 Escalation Process

1. Document issue requiring escalation
2. Contact Privacy Officer with details
3. Privacy Officer provides guidance
4. Implement direction and document

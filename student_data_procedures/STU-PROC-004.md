# Student Data Sharing Procedure (STU-PROC-004)

## 1. Purpose

This procedure provides step-by-step instructions for sharing student data with third parties, including subcontractors, integrations, and legal disclosures, in compliance with STU-POL-004.

## 2. Scope

This procedure applies to all workforce members who facilitate or authorize sharing of student data outside **[Company Name]**.

## 3. Procedure

### 3.1 Subcontractor Engagement

#### 3.1.1 Evaluating New Subcontractors

**When:** Engaging a new subcontractor who will access student data.

**Steps:**

1. **Document business need**
   - What service does subcontractor provide?
   - Is student data access necessary?
   - Can function be performed without student data?

2. **Conduct due diligence**
   - Request privacy policy and data handling practices
   - Request security documentation (SOC 2, ISO 27001)
   - Complete vendor risk assessment
   - Evaluate ability to comply with FERPA/COPPA

3. **Prepare subcontractor agreement**
   - Include required terms (see STU-POL-004 Section 3.3.2):
     - Use limitation
     - No further disclosure
     - Security obligations
     - FERPA compliance
     - Breach notification
     - Data return/destruction
     - Audit rights

4. **Obtain approvals**
   - Security review approval
   - Student Data Privacy Officer approval
   - Legal approval of agreement

5. **Execute agreement**
   - Obtain signatures
   - File in contract management system

6. **Add to subcontractor list**
   - Update internal subcontractor registry
   - Prepare disclosure for school requests

7. **Configure access**
   - Grant minimum necessary access
   - Document access scope
   - Enable audit logging

#### 3.1.2 Annual Subcontractor Review

**When:** Annually for all subcontractors with student data access.

**Steps:**

1. **Generate subcontractor list**
   - All subcontractors with student data access
   - Access scope and data types

2. **Review each subcontractor**
   - Is access still necessary?
   - Are security certifications current?
   - Any incidents or concerns?
   - Agreement terms still adequate?

3. **Update as needed**
   - Renew/update agreements
   - Adjust access if needed
   - Remove access if no longer needed

4. **Document review**
   - Record review date and findings
   - File attestations received

### 3.2 School-Directed Integrations

#### 3.2.1 Processing Integration Requests

**When:** School requests data integration with another system (LMS, SIS, etc.).

**Steps:**

1. **Receive request**
   - Written authorization from school
   - Target system identification
   - Data elements to be shared
   - Purpose of integration

2. **Verify authorization**
   - Request from authorized school contact
   - School confirms they have authority to direct integration

3. **Evaluate integration**
   - Technical security of target system
   - Data minimization (only necessary data)
   - Encryption in transit

4. **Configure integration**
   - Set up secure connection
   - Configure data scope per authorization
   - Enable audit logging

5. **Test and validate**
   - Verify correct data flows
   - Confirm security controls
   - Document configuration

6. **Notify school**
   - Confirm integration active
   - Provide documentation

7. **Monitor ongoing**
   - Log API access
   - Alert on anomalies
   - Review periodically

### 3.3 Legal Disclosures

#### 3.3.1 Responding to Legal Process

**When:** Receiving subpoena, court order, or regulatory request for student data.

**Steps:**

1. **Receive and log**
   - Log receipt date and details
   - Identify issuing authority
   - Identify data requested

2. **Immediately notify Legal**
   - Forward legal process to Legal department
   - Do not respond without Legal guidance

3. **Verify validity** (Legal responsibility)
   - Is process valid and properly issued?
   - Does it apply to **[Company Name]**?
   - Is scope appropriate?

4. **Notify school** (unless prohibited)
   - Inform contracting school of legal process
   - Allow school to object or intervene
   - Document notification

5. **Limit disclosure**
   - Only data specifically required
   - Minimize scope where possible
   - Apply protective order if available

6. **Execute disclosure** (if required)
   - Produce required data
   - Use secure transmission
   - Document what was disclosed

7. **FERPA recordkeeping**
   - Record disclosure in FERPA log
   - Include recipient, data disclosed, date
   - Retain per FERPA requirements

#### 3.3.2 Emergency Disclosures

**When:** Health or safety emergency requiring immediate disclosure.

**Steps:**

1. **Assess emergency**
   - Is there imminent threat to health or safety?
   - Is disclosure necessary to address threat?
   - Who needs the information?

2. **Make disclosure**
   - Disclose minimum necessary information
   - To appropriate parties (law enforcement, school, emergency services)
   - Document decision and rationale

3. **Notify school**
   - Inform school as soon as practicable
   - Provide details of disclosure

4. **Document**
   - Record in disclosure log
   - Include emergency circumstances
   - Retain documentation

### 3.4 Research Disclosures

#### 3.4.1 Processing Research Requests

**When:** Request to share student data for research purposes.

**Steps:**

1. **Evaluate request**
   - Is research authorized in DPA?
   - Does it meet FERPA studies exception?
   - Is IRB approval in place (if required)?

2. **Obtain school authorization**
   - School must authorize research sharing
   - Document school's approval

3. **Prepare data use agreement**
   - Specific research purpose
   - Data security requirements
   - Re-disclosure prohibition
   - Data destruction requirements
   - Publication restrictions (no re-identification)

4. **De-identify if possible**
   - Prefer de-identified data for research
   - If identifiable data required, justify necessity

5. **Execute agreement**
   - Obtain researcher signatures
   - File agreement

6. **Transfer data**
   - Secure transfer method
   - Minimum necessary data
   - Document transfer

7. **Monitor compliance**
   - Periodic check-ins with researcher
   - Verify data destruction at project end

### 3.5 Maintaining Disclosure Records

#### 3.5.1 FERPA Disclosure Log

**When:** Any disclosure of education records to third parties (except schools and routine disclosures).

**Steps:**

1. **Record each disclosure**
   - Name of recipient
   - Recipient's legitimate interest
   - Date of disclosure
   - Description of information disclosed

2. **Maintain log**
   - Centralized disclosure log
   - Organized by student or school
   - Readily accessible for audit

3. **Make available to schools**
   - Provide disclosure log upon school request
   - Schools include in their FERPA records

## 4. Escalation

Escalate to Student Data Privacy Officer when:
- Unusual disclosure request
- Legal process involving student data
- Subcontractor compliance concerns
- School disputes sharing decision
